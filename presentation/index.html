System Overview

What RAID Does

RAID replaces hazardous manual railway inspections with an autonomous IoT robot that continuously scans tracks, detects cracks with IR sensors, captures geo-tagged photographs, and streams real-time alerts to a cloud dashboard — enabling proactive, data-driven maintenance.

Functional Requirements

What the System Must Do

🤖 Robot / Embedded
📡
FR-01 · Autonomous Navigation
Robot traverses the rail line continuously on its own motor drive system
🔍
FR-02 · Multi-Zone IR Scanning
Scans LEFT, CENTER, RIGHT rail zones at 50 ms intervals; requires 6 consecutive positive reads (~300 ms) to confirm a crack
📍
FR-03 · GPS Geo-Tagging
GPS coordinates are "frozen" at crack detection moment for accuracy; published with every alert
📸
FR-04 · Camera Triggering
Master ESP32 triggers the correct zone camera via MQTT command + GPIO pulse (500 ms physics offset for alignment)
🖥️
FR-05 · Local LCD Display
20×4 I²C LCD shows system status, crack zone, and channel number in real time
💓
FR-06 · Heartbeat Publishing
Device sends NOMINAL_HEARTBEAT every 30 s with GPS + IR telemetry when no crack is detected
☁️ Cloud & Application
⬆️
FR-07 · Secure Image Upload
Lambda generates pre-signed S3 PUT URLs; camera uploads JPEG directly to S3 without exposing credentials
🗄️
FR-08 · Persistent Storage
All crack events stored in DynamoDB with SensorID (PK) + timestamp (SK), plus S3 image URL linkage
📊
FR-09 · Real-Time Dashboard
Spring Boot re-broadcasts MQTT data over WebSocket (STOMP); React dashboard updates live without refresh
🗺️
FR-10 · Live Geo-Map
Leaflet.js renders real-time crack markers on a map coloured by severity (RED / AMBER / BLUE)
📄
FR-11 · PDF Report Export
Operators can export Daily/Weekly/Monthly inspection reports as PDF using jsPDF + html2canvas
🔐
FR-12 · Authenticated Access
Supabase auth gates the dashboard; protected routes redirect unauthenticated users to login
Non-Functional Requirements

Dependability, Efficiency, Scalability, Security & UX

🛡️
Dependability
Auto-reconnect: WiFiMulti rotates across 4 APs; MQTT reconnects on disconnect; ESP restarts after 5 failures
Debounce: 6 consecutive IR readings required — eliminates vibration false positives
Camera timeout: 60 s watchdog — publishes alert without image if camera fails
Time sync: NTP (pool.ntp.org) at boot ensures accurate timestamps
STOMP reconnect: 5 s auto-reconnect in frontend keeps dashboard live
⚡
Efficiency
Scan interval: 50 ms sensor polling with interrupt-free main loop
LCD refresh: 250 ms — minimum CPU overhead
MQTT buffer: 1024 bytes — tuned for JSON payload size
Frontend cache: Max 100 live events in memory per sensor stream
Lazy fetch: Historical data loaded once on mount; incremental real-time updates thereafter
📈
Scalability
Fleet support: Dashboard supports N devices (esp-001…N) — DynamoDB partition key per sensor zone
Load balancing: Mission Control shows load distribution across all active robots
S3 storage: Effectively unlimited image storage with no capacity planning
DynamoDB: Auto-scaling; no manual shard management
Mock devices: esp-002 and esp-003 simulate additional robots for testing
🔒
Security
mTLS (device↔cloud): X.509 certificates embedded in firmware; TLS 1.2 on port 8883
mTLS (backend↔cloud): PKCS12 keystore (backend-keystore.p12) loaded at runtime
S3 pre-signed URLs: Camera never holds AWS credentials; URL is the secret
Supabase JWT: Dashboard access protected; JWT verified on every API call
CORS policy: Explicit allowlist in Spring Boot — no wildcard origins
🎨
User Experience
Live status: Animated pulse badges show real-time connectivity of each robot
Severity colour: Map markers RED (≥70%) / AMBER (40-70%) / BLUE (<40%) for quick triage
PDF export: One-click report download — no backend call needed
Responsive: Mobile-first layout with bottom navigation bar
Error graceful: "No GPS" / "No Image (Timeout)" shown instead of blank fields
📊
Maintainability
Layered arch: Clear separation — Firmware / Cloud / Backend / Frontend
DTO pattern: IRSensorDataDTO + EspCamDetectionDTO decouple MQTT from persistence
PlatformIO envs: Separate build targets for each ESP32 module type
Mock simulator: robot_simulator.py enables backend testing without hardware
Open source stack: Arduino, Spring Boot, React — wide community support

• Backend ↔ AWS: PKCS12 keystore + TLS 1.2
• No AWS root credentials on any device
• S3 access only via short-lived pre-signed PUT URLs
• IoT Policies restrict each device's allowed topics
